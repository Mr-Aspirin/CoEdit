### 1. 需求分析概述

CoEdit 是一个基于 Spring Boot 3 和 Vue 3 的多人在线文档协作平台。其核心价值在于支持多用户实时协同编辑同一文档，并提供了完善的权限控制和文档管理功能。

#### 1.1 功能模块拆解
根据 README 的描述，系统主要分为四个功能模块：

1.  **用户管理模块**：负责用户的身份认证与个人信息维护。
    *   核心功能：注册、登录、密保找回密码、个人资料编辑、头像上传。
2.  **文档管理模块**：负责文档的生命周期与组织结构。
    *   核心功能：文档增删改查、文件夹分类、富文本编辑、自动保存、文档搜索（多维度）。
3.  **实时协作模块**：系统的核心，处理多用户交互。
    *   核心功能：多人同时编辑、实时光标显示、评论与@提及。
    *   权限控制（ACL）：分为 Owner（所有者）、Admin（管理员）、Editor（编辑者）、Viewer（查看者）。
4.  **通知与通讯模块**：处理异步消息与用户交互提醒。
    *   核心功能：协作邀请、权限变更、@提及通知、通知列表管理。

### 2. 系统角色定义 (Actors)

在绘制用例图之前，我们需要明确系统中的参与者：

| 角色 (Actor) | 描述 | 权限简述 |
| :--- | :--- | :--- |
| **未注册用户** | 尚未登录系统的访客 | 仅能进行注册或登录操作。 |
| **已登录用户** | 成功登录系统的通用用户 | 可以管理个人信息、创建文档、查看通知、搜索文档。 |
| **文档所有者 (Owner)** | 文档的创建者 | **最高权限**。拥有 Admin 所有权限 + 删除文档 + 权限不可变更。 |
| **文档管理员 (Admin)** | 被授权管理文档的用户 | **管理权限**。拥有 Editor 所有权限 + 管理协作者（增删改权限）+ 删除他人评论。 |
| **文档编辑者 (Editor)** | 被邀请协作编辑的用户 | **编辑权限**。拥有 Viewer 所有权限 + 编辑文档内容。 |
| **文档查看者 (Viewer)** | 仅被授权查看的用户 | **只读权限**。查看文档内容 + 发表评论。 |

### 4. 详细用例表 (Use Case Tables)

选取系统中几个最具代表性的核心场景进行详细描述。

#### 4.1 用例：实时协作编辑 (Real-time Collaboration)

| 项目 | 内容 |
| :--- | :--- |
| **用例名称** | **实时协作编辑文档** |
| **参与者** | 文档所有者、文档管理员、文档编辑者 |
| **前置条件** | 1. 用户已登录。<br>2. 用户已进入文档编辑页面。<br>3. 用户拥有 Editor 及以上权限。 |
| **主事件流** | 1. 用户在富文本编辑器中输入或修改内容。<br>2. 系统通过 WebSocket 实时将变更同步给其他在线协作者。<br>3. 系统在编辑器中实时显示该用户的光标位置和名称标签。<br>4. 系统自动触发保存机制（"一有变更就保存"）。 |
| **备选流** | **网络中断**：若 WebSocket 断开，系统提示“连接中断”，并暂停同步，待重连后尝试合并或提示冲突。 |
| **后置条件** | 文档内容被更新，且所有在线协作者看到的内容一致。 |

#### 4.2 用例：管理协作者 (Manage Collaborators)

| 项目 | 内容 |
| :--- | :--- |
| **用例名称** | **管理协作者与权限 (ACL)** |
| **参与者** | 文档所有者 (Owner)、文档管理员 (Admin) |
| **前置条件** | 1. 用户已登录并打开目标文档。<br>2. 用户具有 Owner 或 Admin 角色。 |
| **主事件流** | 1. 用户点击“协作管理”或“分享”按钮。<br>2. 系统显示当前协作者列表及权限。<br>3. **邀请协作**：用户输入被邀请人用户名/邮箱，选择初始权限（Admin/Editor/Viewer），发送邀请。<br>4. **修改权限**：用户更改现有协作者的角色（例如将 Viewer 升级为 Editor）。<br>5. **移除协作**：用户将某协作者从列表中移除。<br>6. 系统发送相应的通知（INVITE, PERMISSION_UPDATE, REMOVED_FROM_DOC）给受影响的用户。 |
| **业务规则** | 1. **Owner 保护**：任何人都不能修改 Owner 的权限，也不能移除 Owner。<br>2. **Admin 限制**：Admin 可以管理其他协作者，但不能移除 Owner 或修改 Owner 权限。 |
| **后置条件** | 文档的访问控制列表 (ACL) 更新，受影响用户权限即时生效。 |

#### 4.3 用例：文档搜索 (Search Documents)

| 项目 | 内容 |
| :--- | :--- |
| **用例名称** | **文档搜索与排序** |
| **参与者** | 已登录用户 |
| **前置条件** | 用户已登录并进入文档列表页。 |
| **主事件流** | 1. 用户在搜索框输入关键词。<br>2. 用户选择过滤条件（按作者、按日期范围）。<br>3. 用户选择排序方式（按时间倒序/正序、按匹配度）。<br>4. 系统根据条件检索用户有权访问的文档（包括自己创建的和被邀请协作的）。<br>5. 系统展示搜索结果列表。 |
| **后置条件** | 界面展示符合条件的文档列表。 |

#### 4.4 用例：评论与提及 (Comment & Mention)

| 项目 | 内容 |
| :--- | :--- |
| **用例名称** | **发表评论与@提及** |
| **参与者** | 所有具有访问权限的用户 (Owner, Admin, Editor, Viewer) |
| **前置条件** | 用户已打开文档。 |
| **主事件流** | 1. 用户选中一段文档内容或直接点击评论区。<br>2. 用户输入评论内容。<br>3. **@提及**：用户输入 `@` 符号，系统弹出协作者列表，用户选择要提醒的人。<br>4. 用户提交评论。<br>5. 系统在文档对应位置标记评论，并向被 `@` 的用户发送 MENTION 类型的通知。 |
| **后置条件** | 评论显示在文档侧边栏或对应位置，被提及用户收到通知。 |

### 5. 总结

CoEdit 项目的需求清晰地指向了一个**重交互、重权限**的即时通讯类应用。在开发过程中，后端需重点关注 WebSocket 的并发处理和 ACL 权限拦截逻辑，前端需重点处理 Quill 编辑器的协同状态同步（Operational Transformation 或 CRDT 算法集成）以及光标的实时渲染。
